שרותי ממערכת הפעלה - א 📊

operating system services

🔄 ריבוי תכנותי (Multi Programming)

ריבוי תכנותי בגרסאות ישנות של מערכות (Batch Systems) מאפשר לשמור מספר עבודות בזיכרון כך שה-CPU לא יעמוד ללא עבודה. המערכת בוחרת עבודה להפעלה, וכאשר היא ממתינה לקלט/פלט, עוברת להריץ עבודה אחרת.

נקודות חשובות:
  • מדמה ריבוי למרות שבפועל זמן המעבד מתחלק בין עבודות בזמנים גדולים.
  • מתאים בעיקר למערכות אצווה (Batch).
  • אם עבודה אחת מחכה לקלט/פלט, המעבד עובר לעבודה אחרת.

⏱️ ריבוי משימות (Multi Tasking / Time Sharing)

ריבוי משימות מאפשר חלוקת זמן המעבד בין תהליכים בצורה דינמית וקצרה, כך שהמשתמש מרגיש תגובה מיידית לאינטראקציה. מתאים למערכות אינטראקטיביות.

  • חלוקת זמן קצרה (Time Slice): כל תהליך מקבל קצבת זמן קצרה של המעבד.
  • תגובה מהירה: המשתמש יכול לראות תגובה כמעט מיידית.
  • הקצאת CPU דינמית: מערכת ההפעלה בוחרת תהליך הבא על בסיס נימוקים שונים (עדיפויות, מעקב זמן ריצה וכדומה).

    • ⚙️ פעולה במצב ריבוי (Dual-mode Operation)

    מצב כפול (User Mode ו-Kernel Mode) הוא חלק מהגנה: קריאות מערכת מאפשרות מעבר ממצב משתמש לגרעין, והרשאה למבצעים רגישים. במצב משתמש פעולות מסוימות נחסמות, ורק דרך system call המעבד מועבר למצב גרעין.

    // דוגמה: מעבר ממצב משתמש לגרעין בקריאת מערכת
    system_call(…); // המעבד עובר למצב גרעין לביצוע פעולה רגישת
    // לאחר סיום הפעולה, המעבד חוזר למצב משתמש באופן בטוח.
    הערה: המעבר נעשה תמיד דרך מנגנון מוגן (trap), לא באמצעות קפיצה ישירה, מה שמונע הזרקת קוד זדוני למצב גרעין.

    ⏲️ שעון עצר (Timer Interrupt)

    המערכת מגדירה טיימר שמקטין ערך בכל יחידת זמן. כאשר הטיימר מתאפס, נשלחת פסיקה (Interrupt) לגרעין, המאפשר לו לבצע preemption ולבחור תהליך אחר לריצה.

  • מניעת לולאות אינסופיות: אם תהליך נהיה תקוע, הטיימר מאפשר למערכת לחזור ולהחליט שלא להמשיך בתהליך הבעייתי.
  • preemption: מאפשר למערכת לחתוך ריצה של תהליך ולרוץ בתהליך אחר לפי מדיניות התזמון.
  • הגדרת קצבת זמן: כל תהליך מקבל קצבת CPU קצובה מראש, ושעון העצירה מבטיח חזרה לגרעין בתום הקצבה.

    • 🔧 ניהול תהליכים (Process Management)

    תהליך הוא יישות אקטיבית: מסגרת להרצת תוכנה עם מצבים (state), סגמנטים בזיכרון ומשאבים שונים. מערכת ההפעלה מטפלת ביצירה, השהייה, המשך, והרג של תהליכים.

    📦 תהליך
    ↓
    ⚙️ קוד + נתונים בזיכרון
    ↓
    🖥️ משאבי מערכת (CPU, I/O)

    ⚙️ פעילות ניהול תהליכים (Process Management Activities)

    • יצירה והרס תהליכים: fork/spawn והקצאת משאבים ראשונית; הריגה ושחרור משאבים.
    • תזמון ותעדוף: בחירת תהליך לריצה לפי מדיניות (FCFS, Round Robin, Priority וכו').
    • השעיה והמשך: עצירת תהליך (לדוגמה: המתנה ל-I/O) והמשך ריצה כשהמשאב מוכן.
    • סינכרון: מנגנונים למניעת תחרות (locks, semaphores) ולטיפול ב-deadlock.
    • תקשורת בין תהליכים (IPC): העברת הודעות, shared memory, pipes וכו'.
    • טיפול במבוי סתום (Deadlock): זיהוי, מניעה, טיפול והשבתת תהליכים במידת הצורך.

    🧠 ניהול זיכרון (Memory Management)

    מערכת ההפעלה קובעת איזה חלק בזיכרון מוקצה לתהליך ומתי, עוקבת אחרי מיקום וזמינות, תומכת ב-virtual memory ושחלוף (swapping) במידת הצורך.

    נקודות עיקריות:
    • מעקב אחר משתמשי הזיכרון: מה תפוס ומה פנוי.
    • הקצאה דינמית: פרוסת זיכרון לכל תהליך לפי דרישה.
    • עיבוי ושחרור: שחלוף תהליכים לדיסק (swap) אם אין מקום בזיכרון הראשי.
    • Virtual Memory: מאפשר להריץ תהליכים חלקית בזיכרון הראשי, עם pages ו-TLB.

    📁 ניהול מערכות קבצים (File System Management)

    מערכת ההפעלה מספקת מבנה לוגי אחיד (קבצים, ספריות) מעל אחסון פיזי שונה, מטפלת ביצירת/מחיקה, הקצאה ומיפוי לדיסק, בקרת גישה ובטיחות.

    • יצירה ומחיקה: קבצים וספריות עם הרשאות מתאימות.
    • מיפוי לדיסק: חלוקת בלוקים/inodes ואלגוריתמים לניהול חלל פנוי.
    • תזמון דיסק: סדר פעולות קריאה/כתיבה לשיפור ביצועים.
    • גיבוי ושחזור: מנגנונים לשמירת נתונים ארוכי טווח והחלמת תקלות.
    • בקרה והרשאות: קביעת מי יכול לקרוא/לכתוב/להריץ קבצים.

    💽 ניהול אחסון המוני (Mass-Storage Management)

    טיפול בהתקני אחסון שונים (HDD, SSD, רשת). התאמה ליכולות שונות של מדיומים (קיבולת, מהירות, אופן גישה). מערכת ההפעלה מספקת ממשק אחיד ובחירה באלגוריתמים מתאימים.

    Cache/Buffer
    ↓
    RAM
    ↓
    SSD/HDD/Network Storage

    ⚡ מטמון (Caching)

    זיכרון מהיר קטן המשמש לאחסון מידע או נתונים שנטענים בתדירות גבוהה, במטרה לקצר זמני גישה ולהפחית תעבורה אל הזיכרון הראשי או אחסון איטי.

  • Cache Levels: L1, L2, L3 – קרבה ומהירות שונה.
  • מדיניות החלפה: LRU, FIFO וכדומה.
  • Cache Coherence: נושא חשוב בסביבות רב-מעבד ורב-ליבות.

    • 🔌 מערכת קלט/פלט (I/O Subsystem)

    מערכת ההפעלה מנהלת התקנים בעזרת דרייברים, מפשטת גישה אחידה, מטפלת缓冲 (buffering), מטמון, סלילים (spooling) וטיפול ב-interrupts.

    • דרייברים: קוד המתווך בין גרעין להתקן ספציפי.
    • Buffering & Spooling: אחסון ביניים ושילוב פעולות I/O לשיפור ביצועים.
    • Interrupt Handling: תגובה מהירה לאירועי חומרה.
    • ממשק אחיד: API אחיד לתוכנות מעבר לסוג ההתקן.

    🔒 אבטחה והגנה (Protection & Security)

    מנגנונים למניעת גישה בלתי מורשית למשאבים ולתהליכים, וכן הגנה מפני התקפות פנימיות וחיצוניות.

  • בקרת גישה: מנגנוני הרשאות, משתמשים וקבוצות.
  • בידוד תהליכים: מניעת גישה אל זיכרון תהליכים אחרים.
  • הגנת המערכת: antivirus, firewalls, מניעת DoS.
  • Authentication/Authorization: אימות וזהוי לפני גישה למשאבים רגישים.

    • 🌐 סביבת מחשב (Computing Environments)

    הקשרים שבהם פועלת מערכת ההפעלה: וירטואליזציה, מערכות מבוזרות, ענן, mobile וכו'. כל סביבה מציבה דרישות משלה לניהול תהליכים ומשאבים.

    וירטואליזציה (Virtualization)

    טכניקה המאפשרת להריץ מספר מערכות הפעלה על אותה חומרה פיזית, תוך יצירת מכונות וירטואליות (VMs) מבודדות.

    • מנהל מכונות וירטואלי (VMM/Hypervisor): מספק שירותי וירטואליזציה ומאפשר למערכות הפעלה לרוץ במקביל, כמו VMware ESX או Citrix Xenserver.
    • וירטואליזציה מלאה: הרצת מערכת הפעלה שלמה בסביבה מדומה, ללא שינויים בקוד המקורי.
    • וירטואליזציה חלקית: שינויים קלים במערכת ההפעלה לשיפור ביצועים במכונה הוירטואלית.

    אמולציה (Emulation)

    משמשת כאשר מעבד המקור מסוג שונה מהיעד (למשל PowerPC לאינטל x86). זוהי השיטה האיטית ביותר בדרך כלל.

    // אמולציה מתרגמת כל פקודה בזמן אמת
    for each instruction_from_source_architecture:
        translate_and_execute(instruction) // תרגום ותהליך איטי

    פרשנות (Interpretation)

    כאשר שפת מחשב לא נערכה לקוד המקורי - כמו מכונות וירטואליות של Java (JVM).

    דוגמאות סביבות מחשוב:
    • מארחי VM וקונטיינרים (Docker, Kubernetes)
    • מחשוב מקבילי ומבוזר (Cluster Computing)
    • שירותי ענן (AWS, Azure, GCP)
    • מערכות משובצות (Embedded Systems)
    • מערכות זמן אמת (Real-time Systems)

    🏗️ ארכיטקטורות מחשב (Computer Architectures)

    מבנה וארגון המעבדים והרכיבים: מעבד יחיד, רב-מעבד (multi-processor), רב-ליבות (multi-core), GPU ועוד. ארכיטקטורה משפיעה על ניהול תהליכים, סינכרוניזציה וזיכרון משותף.

  • Single Processor: כל תהליך משתף זמן על מעבד אחד.
  • Multi-Processor: מספר מעבדים נפרדים; דורש סינכרוניזציה ושמירה על קוהרנטיות זיכרון.
  • Multi-Core: ליבות מרובות במעבד יחיד עם מטמון משותף או נפרד.

    • ⚖️ רב מעבד מול רב ליבות (Multi-Processor vs Multi-Core)

    בד"כ ההבדל בינהם הוא ברמות המטמון ובקוהרנטיות:

    • מטמון נפרד לכל יחידה: עלול לגרום לאי-קוהרנטיות ולתקשורת מרובה.
    • מטמון משותף: פותר חלק מבעיות הקוהרנטיות אך דורש מנגנוני סנכרון מורכבים.
    • ניהול עומס: חלוקת תהליכים/threads בין מעבדים או ליבות.
    • יעילות אנרגטית וביצועים: ליבות בתוך שבב נוטות לחיסכון אנרגיה, אך אתגר בניהול משאבים משותפים.

    🧩 גישה לא אחידה לזיכרון (NUMA - Non-Uniform Memory Access)

    במערכות with NUMA, לכל יחידת עיבוד יש גישה מהירה לבלוקים מסוימים של זיכרון וגלויה יותר לאחרים. יתרון: קריאה/כתיבה ל"זיכרון קרוב" מהירה; חסרון: אם תהליך רץ ב"צומת" לא קרוב לזיכרון שלו - ביצועים יורדים.

  • יתרונות: שיפור ביצועים כשמטפלים בחישובים מקומיים בזיכרון.
  • חסרונות:
    • גישה לזיכרון מרוחק היא איטית יותר.
    • אתגרים בהמשכיות תהליכים.
    • אי-איזון עומסים בין מעבדים.

    • 🏛️ סביבה מסורתית (Traditional Environment)

    בסביבה מסורתית, מערכת ההפעלה רצה על חומרה ייעודית או שרת מקומי, עם משאבים מוגדרים וגלויים. מרבית התהליכים והשירותים פועלים במכונה יחידה או בשרת פנימי.

    • משאבים קבועים: כמות זיכרון, CPU ודיסק ידועים מראש ואינם משתנים בזמן ריצה.
    • ניהול ריכוזי: כל השירותים והתהליכים מנוהלים על ידי מערכת הפעלה אחת מרכזית.
    • תחזוקה ידנית: שדרוגים והתקנות מתבצעים באופן ידני או עם כלי ניהול פנימיים.
    הערה: נוח לתשתיות קטנות או מערכות פנימיות שאינן דורשות סקלאביליות דינמית.

    📱 סביבה ניידת (Mobile Environment)

    סביבה שבה מערכת ההפעלה מותאמת למכשירים ניידים (טלפונים, טאבלטים). משאבים מוגבלים יותר, דרישות חיסכון באנרגיה, וניהול חיבורים משתנים.

    • ניהול אנרגיה: הימנעות מתהליכים כבדים ברקע, שינה (sleep) והתקדמות למצב נמוך כדי לחסוך סוללה.
    • חיבורים דינמיים: Wi-Fi, סלולרי ושינויים בתנאי רשת המחייבים התאמה דינמית של שירותי רקע.
    • אבטחה מובנית: sandboxing לאפליקציות, אישורי גישה למשאבים (מצלמה, מיקום), והגנה בפני התקפות נפוצות בסלולרי.

    💻🖥️ סביבה שרת-לקוח (Client-Server Environment)

    בארכיטקטורת שרת-לקוח, התהליכים מתחלקים בין לקוחות הפונים לשרת מרכזי שמספק שירותים או נתונים.

    • הקצאת משאבים בשרת: השרת מנהל תורים של בקשות, thread pools, balancing למשאבים.
    • סקלאביליות: הוספת שרתים או הרחבת כוח עיבוד לפי עומסים.
    • תקשורת תקינה: פרוטוקולים (HTTP, RPC) וטיפול ב-latency, טעינת timeout וניהול שגיאות.
    • אבטחה: אימות משתמשים, הצפנה בתעבורת הנתונים, בידוד בין לקוחות.
    הערה: נדרש ניהול עומסים (Load Balancing), caching מצד הלקוח והשרת, ושירותי ניטור לביצועים.

    🔗 סביבה עמית לעמית (Peer-to-Peer Environment)

    ב-P2P כל יחידה (node) פועלת גם כלקוח וגם כשרת, משתפת משאבים ישירות עם עמיתים אחרים ללא שרת מרכזי בלעדי.

    • חיפוש ומציאה: פרוטוקולים מבוזרים למציאת משאבים או שירותים בקרב עמיתים.
    • הפצה מבוזרת: הפצת קבצים או נתונים באופן מבוזר (למשל torrent) כשכל node מעביר לחבריו.
    • עמידות לתקלה: אין נקודת כשל מרכזית; אם node מתנתק, הרשת ממשיכה בתפקוד.
    • בעיות אבטחה: אימות זהויות בקשה מורכבת יותר, חשש מהפצת תוכן לא רצוי או זדוני.

    ☁️ מחשוב ענן (Cloud Computing Environment)

    בסביבת ענן המשאבים (Compute, Storage, Network) מסופקים כשרות (IaaS/PaaS/SaaS) דרך האינטרנט, עם סקלאביליות גמישה ותשלום על פי שימוש.

    • Elasticity: הגדלה והקטנה אוטומטית של משאבים בהתאם לעומס.
    • Multi-Tenancy: שרתים ווירטואליים המארחים מספר לקוחות בנפרד, בידוד משאבים ואבטחה.
    • מיקרו-שירותים: ארכיטקטורה מונחת שירותים קטנים הניתנים לפריסה עצמאית בענן.
    • תשתית ממומשת: שימוש ב-VM, containers, serverless; ניהול קונפיגורציה ואוטומציה (IaC).
    נקודות למערכת הפעלה: תמיכה בווירטואליזציה, ניהול משאבים דינמי, תאימות ל-cloud APIs, ושימור קוהרנטיות בביצוע פעולות מבוזרות.

    ⏱️ מערכת משובצת בזמן אמת (Real-Time Embedded System)

    סביבה שבה מערכת ההפעלה מותאמת למכשירים משובצים עם דרישות זמן תגובה קפדניות (hard/soft real-time), ומשאבים מוגבלים מאוד.

    • קפדנות בזמן: תהליכים חייבים להשלים פעולות תוך מגבלה קבועה (deadlines).
    • ניהול משאבים מוקפד: זיכרון, CPU, I/O מוקצים מראש, לרוב ללא swapping.
    • Scheduler מיוחד: ממשקי תזמון קפדניים (Rate Monotonic, Earliest Deadline First) כדי להבטיח עמידה בזמן.
    • אבטחה ובטיחות: קריטית בתחומים רפואיים, תעופה, תעשייה – מערכות חייבות להיות אמינות וללא crash.
    הערה: ל-callbacks ו-interrupts זמן תגובה מינימלי; המערכת לרוב מונוליתית או microkernel מותאם real-time.
